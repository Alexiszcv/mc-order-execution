# Strategy sweep — options & trade-offs (Stream D)

Generated by `scripts/sweep_fill_rate.py` (Pareto) and `scripts/sweep_chase.py` (execution
variants). All numbers are the **v2 no-lookahead** backtest at τ=5, M=N=K=3, half-life=20,
j_start=200. Slippage is in ticks vs the window open; **+ = strategy beats a market order at
the open**. p05 = 5th-percentile slippage (the adverse tail we want to shrink).

**This is a menu, not a recommendation.** Picking τ / target / chase policy is the user's call
(CLAUDE.md). Below is what each knob buys you.

## 1. Fill-rate target sweep (Pareto curves)

Figures: `reports/figures/pareto_fill_rate_<market>.png` (Gold, Nasdaq, Bunds, EuroStoxx, GBP).

Raising `fill_rate_target` always raises the achieved fill rate and **shrinks the tail**, but
**shrinks the median edge** (limits sit closer to the open). The mean barely moves — the chase
tail dominates it. There is **no universal sweet spot**; it differs per market:

| Market (tick) | mean slippage range over target 0.4→0.8 | median 0.4→0.8 | tail p05 0.4→0.8 | comment |
|---|---|---|---|---|
| Gold (0.10) | +0.04 … +0.22 (flat) | +3 → +1 | -16 → -9 | smooth trade; 0.6 fine |
| Nasdaq (0.25) | best (least −ve) at **low** target | +8 → +3 | **-52 → -26** | mean vs tail conflict |
| Bunds (0.01) | peaks ~**0.6–0.7** (+0.25) | +1 → +1 | -8 → -4 | robust; sell@0.7 looks good |
| EuroStoxx (1.0) | peaks ~**0.5–0.6** (+0.29) | +2 → 0 | -6 → 0 | edge gone by 0.8 (96% fill) |
| GBP (0.0001) | +0.8 … +5.6 | **+100 … +200** | **-600 … -900** | pathological — see caveat |

Takeaways for the user to weigh:
- If you optimise **median / typical fill**, a lower target (0.4–0.6) keeps a bigger edge.
- If you optimise **tail risk**, a higher target (0.7–0.8) roughly halves the worst-case.
- **Nasdaq** is the clearest conflict: lowest target gives the least-negative *mean*, highest
  target gives the smallest *tail*. You can't have both with this single knob.
- **0.6** (the current default) is defensible for Gold/EuroStoxx; Bunds leans 0.7; Nasdaq
  depends entirely on whether you care about mean or tail.

> **GBP caveat:** medians of +100–200 ticks and tails of -600–900 ticks come from the 0.0001
> tick making 5-min ranges hundreds of ticks wide. The tail magnitude (≈7% moves in 5 min) is
> implausible and smells like roll/data-quality artifacts. **Do not trust GBP numbers until
> Stream B (data/liquidity) has vetted the contract rolls and outlier bars.**

## 2. Execution / chase variants (decision path held fixed)

From `scripts/sweep_chase.py`. The picker + regime path are identical to baseline; only the
**unfill handling** changes (except cost-aware, which changes the picker). `baseline` is
asserted to reproduce `run_backtest_rolling` exactly, so these are apples-to-apples.

Representative numbers (sell side; buy is similar):

| Variant | Gold avg / p05 | Nasdaq avg / p05 | Bunds avg / p05 | fill rate | what it does |
|---|---|---|---|---|---|
| baseline (chase@close) | +0.18 / -13 | +0.12 / -42 | +0.21 / -6 | 66–74% | chase at the window close |
| **mid-chase** | +0.86 / -9 | **+2.16 / -28** | +0.51 / -4 | **same as baseline** | chase at window mid (H+L)/2 |
| **early-chase (>4t)** | +0.37 / **-4** | **+2.94 / -4** | +0.11 / -4 | 53–67% | bail intrabar at the trigger |
| cost-aware (c=2t) | +0.19 / -15 | +0.22 / **-54** | +0.16 / -6 | 52–70% | expected-cost picker, chase@close |

What each buys you:
- **mid-chase** — strictly better mean *and* tail than baseline at the **same fill rate**, on
  every market. The catch: it models the unfilled order as getting done at the window mid,
  which assumes you can work a marketable order to ~mid rather than paying the close. If that
  assumption holds operationally, this is the cheapest win on the board.
- **early-chase (trigger=4 ticks)** — **caps the tail** hard (p05 collapses to ~-4, the trigger
  level) and on the volatile market (Nasdaq) it actually *lifts* the mean a lot, because cutting
  the loss early beats riding to the close. Cost: fill rate drops (~53–67%) and the median edge
  shrinks to the trigger. This is the tool when **tail risk** is the priority. `trigger_ticks`
  is the knob (smaller = tighter tail, lower fill).
- **cost-aware picker (chase_cost=2t)** — an alternative to a fixed target: it picks ℓ* to
  maximise `P(fill)·ℓ − P(unfill)·chase_cost`. At c=2 it places *wider* limits than the 0.6
  target, so fill drops and the tail gets *worse* (it misses bigger). It's not a free win at
  c=2; it needs calibration — a larger `chase_cost_ticks` pulls it conservative (toward small ℓ,
  high fill). Surfaced as a knob, not a recommendation.

## How to reproduce

```bash
python scripts/sweep_fill_rate.py   # Pareto curves + table (5 markets, ~90s)
python scripts/sweep_chase.py       # chase/picker comparison + table (3 markets)
pytest -q tests/test_strategy.py    # picker + chase-model unit tests
```

## Open items for the user to decide
1. Which objective matters — median edge, mean, tail, or fill? The answer picks the target.
2. Is the **mid-chase** fill model (work-to-mid on unfill) operationally realistic? If yes, adopt it.
3. Tail-averse? Set an **early-chase** `trigger_ticks` and accept the lower fill.
4. GBP (and any other tiny-tick FX) needs a Stream-B data sanity pass before its numbers count.
